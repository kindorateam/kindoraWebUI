# Kindora API Implementation Audit

This report compares `../api-main-ts` with the product requirements in `product-implementation-roadmap.md` and the API contracts currently used by `kindoraWeb`.

The audit is based on routes, controllers, Prisma models, authorization, tests, and static code analysis. A registered route is not treated as a complete product capability when its required history, auditability, delivery, or compliance behavior is missing.

## Status legend

- ✅ Implemented and database-backed
- 🟡 Partially implemented or blocked by a contract/security gap
- ⬜ No corresponding API
- 🔴 Implemented unsafely or with a release-blocking defect

## Executive summary

The backend contains **93 registered HTTP/WebSocket handlers** across authentication, companies, employees, students, rooms, conversations, storage, and system health.

It is substantially more complete than the frontend currently exposes:

- Employee creation, invitations, absences, and documents are database-backed; staff documents are now connected in the frontend.
- Conversations, cursor-paginated message history, search, favorites, pins, read cursors, mute state, export, WebSockets, and optional Redis fan-out are implemented. The frontend now uses conversation listing, upward history loading, sending, favorites, read state, and WebSocket updates.
- Student and employee room presence is supported.
- Server-side role authorization and tenant-scoped resource controllers exist for employees, students, rooms, and conversations.

However, the API is still an **administrative platform core**, not the Georgia compliance and cash-flow MVP. There are no API domains for historical attendance, Georgia regulations, inspection packets, incidents, medication administration, policies, drills, menus, billing, payments, CAPS, or migration.

There are also release-blocking security and contract problems in public registration, company tenancy, OAuth state handling, and frontend/backend payload alignment.

## Release-blocking findings

### 🔴 Public employee creation accepts an arbitrary company ID

`POST /api/v1/auth/register` and `POST /api/v1/auth/create` are public and create an employee in any supplied `companyId`. The `/create` route is described as admin creation but has no authentication or authorization.

**Required fix:** remove or tightly constrain public registration, make admin creation use the protected employee endpoint, and use an invitation or signed enrollment token when public onboarding is required.

### 🔴 Company CRUD is cross-tenant

Company routes authorize by role, but company controller reads and writes by global company ID without applying `tenantWhere(user)`. An administrator receives `companies:*`, so a known foreign company UUID can be read, updated, or deleted.

**Required fix:** company access must be scoped to `currentUser.companyId`; keep any platform-level administration in a separate, explicitly controlled trust boundary.

### 🔴 OAuth state is generated but never verified

The OAuth initialization route sets a state cookie, and `verifyOAuthState` exists, but the callback does not compare the query state with the cookie. Its Swagger summary incorrectly claims state verification.

**Required fix:** verify and consume OAuth state for redirect flows. If the SPA authorization-code popup intentionally uses a different contract, separate that route and document its CSRF protections explicitly.

### 🔴 Attendance history is destroyed on checkout

`RoomPresence` stores only `roomId`, `userId`, `userType`, and `startTime`. Check-in deletes any previous presence, and checkout deletes the active record. This supports a live room roster but cannot support Georgia attendance retention, corrections, CAPS reconciliation, or inspection evidence.

**Required fix:** introduce immutable attendance events/sessions with arrival, departure, actor, guardian acknowledgment, source device, correction history, and idempotency metadata. Keep live presence as a projection, not the legal record.

### 🟠 Verification and reset messages are not delivered

Email and SMS functions only write verification codes to stdout. Invitations and password reset appear successful but cannot reach users outside local development.

**Required fix:** add a production notification provider, delivery status, retries, safe logging, and environment-specific development capture.

### 🟠 Resource APIs lack meaningful test coverage

Current tests cover a small OAuth slice, the chat hub, and health behavior. There are no integration tests for tenant isolation, authorization matrices, employee/student/room CRUD, documents, presence, or conversations.

**Required fix:** add database integration tests around tenant boundaries and the high-risk mutation paths before expanding compliance functionality.

## Recommended implementation sequence

1. Fix public registration, company tenancy, OAuth state verification, and production notification delivery.
2. Align frontend/backend roles and the student/employee payload contracts.
3. Complete the remaining employee-document metadata editing and conversation creation/search actions.
4. Add organization locations and an immutable audit-event foundation.
5. Replace current-only presence with auditable attendance sessions/events and an atomic room-transfer API.
6. Add parent/guardian, pickup authorization, child medical, immunization, consent, and completeness APIs.
7. Add the versioned Georgia rules and ratio engine.
8. Add incidents, medication administration, policies, drills, menus, and inspection exports.
9. Add billing/payments, then CAPS reconciliation.

The first implementation step should therefore be a **security and contract-alignment milestone**, before adding new compliance domains.
